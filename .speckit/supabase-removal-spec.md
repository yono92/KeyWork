# Supabase 제거 및 로컬 전용 전환 명세

## 목표

KeyWork에서 Supabase Auth, Database, Realtime, 관리자 service role 의존을 완전히 제거하고, 한/영 싱글 플레이 타이핑 게임이 외부 백엔드 없이 동작하도록 전환한다.

## 사용자 시나리오

### P1 — 백엔드 없이 게임 실행

- 사용자는 Supabase 환경변수가 없어도 모든 싱글 플레이 게임에 진입하고 플레이할 수 있다.
- 모든 페이지 요청에서 Supabase 인증/세션 네트워크 요청이 발생하지 않는다.

### P1 — 로컬 커스텀 문장

- 사용자는 한글/영문 커스텀 연습 문장을 추가·수정·삭제할 수 있다.
- 저장한 문장은 같은 브라우저에서 새로고침 후 유지된다.

### P2 — 로컬 기록

- 온라인 순위 대신 이 브라우저에 저장된 게임 기록과 최고 기록을 확인할 수 있다.

### P1 — 온라인 전용 기능 종료

- 로그인, 프로필, 아바타, 업적, 친구, 게임 초대, 공개 방, 빠른 매칭, 온라인 대전, 관리자 화면은 노출되지 않는다.
- 제거된 라우트와 API는 빌드 산출물에 포함되지 않는다.

## 기능 요구사항

- FR-01: `@supabase/ssr`, `@supabase/supabase-js` 런타임 의존이 없어야 한다.
- FR-02: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`가 없어야 한다.
- FR-03: 브라우저 저장소 접근은 Client Component의 effect 또는 사용자 이벤트 안에서만 수행한다.
- FR-04: 로컬 데이터는 버전이 포함된 키와 런타임 검증/기본값을 사용한다.
- FR-05: Supabase 전용 타입은 도메인 타입으로 이동한다.
- FR-06: 기본 게임과 로컬 기록 라우트는 유지하고 profile/battle/admin/API 라우트는 제거한다.
- FR-07: 한국어와 영어 UI 및 타이핑 동작을 유지한다.

## 비기능 요구사항

- 외부 백엔드 장애와 무관하게 초기 화면과 싱글 플레이가 렌더링되어야 한다.
- 저장소 데이터가 손상되어도 앱은 기본값으로 복구되어야 한다.
- 기존 Win98/Mac-classic 디자인 토큰과 Tailwind 컴포넌트 패턴을 유지한다.
- localStorage는 사용자 브라우저에만 존재하며 기기 간 동기화되지 않음을 UI에 명시한다.

## 범위 제외

- 다른 Auth/DB/WebSocket 서비스 도입
- 기존 Supabase 사용자 데이터 자동 마이그레이션
- P2P/WebRTC 멀티플레이어
- 클라우드 동기화와 글로벌 랭킹

## 성공 기준

- `rg` 기준 Supabase SDK, 환경변수, 클라이언트 생성 코드가 0건이다.
- 싱글 플레이 라우트 5종이 빌드된다.
- 로컬 커스텀 문장과 점수 저장 테스트가 통과한다.
- lint, unit test, production build가 통과한다.
